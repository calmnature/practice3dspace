## Goal
Identify the conditions in the Users Query.

## Context
Condition is a mini query composing of 3 things:
1. tag (property)
2. value (value against the property)
3. operator (connector between tag and value)

Structure of condition: Where tags inside square bracket, followed by operator, followed by value.
This "operator" depends on the type of information:
- for numerical, it can be "<", "<=", ">", ">=", ":"
- for date, it can be "<", ">", ":", "<=", ">="
- for string & boolean, it is simply ":"

** TimeLine management
For timeline, consider replying the equivalents:
- for day(d), week(w), month(m), quarter(q), half(h), year(y)
  - today -> @@d@@
  - yesterday -> @@d-1@@
  - 10 days ago -> @@d-10@@
  - 10 days into future -> @@d+10@@
  - this year -> @@y@@
  - last year -> @@y-1@@
  - 2 years ago -> @@y-2@@
  - this month -> @@m@@
  - next month -> @@m+1@@
  - this week -> @@w@@
  - last week -> @@w-1@@
  - this quarter -> @@q@@
  - last quarter -> @@q-1@@
  - first half -> @@h1@@
  - second half -> @@h2@@
  - 2nd June 2023 -> @@d2m6y2023@@
  - 31st March 2040 -> @@d31m3y2040@@
  - January 2030 -> @@m1y2030@@
  - April -> @@m4@@
  - week 23 -> @@w23@@
  - week 43 of year 2030 -> @@w43y2030@@
  - h1 2023 -> @@h1y2023@@
  - q2 of 2020 -> @@q2y2020@@
  - q4 -> @@q4@@

** ME keyword
For ME context (like "created by me", "owned by me", "my"), use @@me@@

** Structure of Input **
Input object consists of
1. User query
2. Type
3. Kind
// where {Type} & {Kind} fields give "Type of object" information.

### Sample input
if available tags = [Title, Likes, Owner, Created, Modification Date, Collaborative Space, Änderungshistorie, Is Last Revision, LooPs, Loop, Community, Created By, GodSpeedThunder bird, GodSpeedThunder]

{
  User Query: find physical product with bead fastener extension godSpeedThunder 14mm/s created today,
  Type: [Physical Product],
  Kind: [Bead Fastener]
}

## Steps ##
Step 1: Query decomposition
- Break the query into chunks from the available tags & Object-type info.
- Examine every tag, even after finding a match, to see if a better one exists
- Chunks: 'find physical product', 'with bead fastener extension', 'godSpeedThunder 14mm/s', 'created today'
Step 2: Dereferencing "object type" info from User Query
- get rid of chunks that capture "type"/"kind" field information.
- USE Case insensitive phrase match for this step.
- Dereferenced Chunks: 'godSpeedThunder 14mm/s', 'created today'
Step 3: Building conditions
 - for the dereferenced chunks build conditions
 - Maintain the exact form present in {tags} list, char by char.
 - Examine every tag, even after finding a match, to see if a better one exists
 - don't match by Synonymy meaning of the tag, Match by phrase not by meaning of the tag.
 - if NO tag matches the chunk, skip {building conditions} step for that Chunk.
  Conditions: '[GodSpeedThunder]:14', '[Created]:@@d@@'
Note: if everything is already captured in the input, return [].

## Hypothetical Example
If available tags = ['Shut thE door', 'Open window']
Query: shut the door HEllo
Types: []
Explanation: identifying is critical, even when the tag is "too generic" or "absurd". NO mater what , Always Maintain the Same form (capitalization) as in {tags}.
Conditions:
'[Shut thE door]:"HEllo"'

## Rules (Strictly)
  - Ignore Chunks that match entries in TYPE/KIND field. KEEP Partial matches, ONLY IGNORE EXACT MATCHES(Plural forms as well). Check by phrase matching, ignore case.
  - Conditions is a Array of Key/value pairs.
  - In decomposition & Building conditions steps, strictly check every tag, even after finding a match, to see if a better one exists (Case Insensitive phrase match)
  - # Plain‑text‑only mode (do NOT remove)
    - You must output **exactly one plain‑text token** and nothing else.
    - Do NOT wrap the answer in markdown, code fences, JSON, XML, or any other container.
    - Do NOT add a trailing newline, spaces, or any explanatory text.
